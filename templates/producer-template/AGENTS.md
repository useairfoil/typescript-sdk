# Producer Template Agent Notes

Use this file when adapting `templates/producer-template` into a new producer or
when keeping the template aligned with current connector-kit patterns.

The template is intentionally provider-neutral. New connectors copied from it
must add their own connector-local `AGENTS.md` with provider-wide facts for future
debugging and upgrades, not just the first implemented scope.

## Template Purpose

- Demonstrates the canonical producer package shape for this repo.
- Uses JSONPlaceholder so tests can run without external credentials.
- Shows one REST-style entity, paginated backfill, queue-backed webhook events,
  `Webhook.route(...)`, VCR replay, and in-memory webhook tests.
- Should remain simple and generic. Do not add provider-specific behavior here.

## Provider Research Checklist

Before adapting this template, collect and record:

- API docs and changelog URLs, with retrieval date.
- API versioning policy, selected version, deprecation window, and how to detect
  served version drift.
- Auth scheme, token type, dashboard setup path, required scopes, approval gates,
  token rotation/expiry behavior, and secret handling.
- Provider-maintained SDK, if any: package name, current version, changelog URL,
  what contract it handles, such as webhook signing, generated types, pagination,
  or API clients, and whether the connector should pin it explicitly.
- Sandbox or test-mode base URLs, test-data requirements, and production/sandbox
  separation.
- Pagination style, default and maximum page sizes, continuation token/cursor
  rules, response fields, and rate limits.
- Webhook support, event catalog, delivery timeouts, retry policy, idempotency
  keys, signature headers, canonical signing payload, raw-body requirements, and
  timestamp tolerance if documented.
- Platform resource map for likely future entities, not just the first entities
  being implemented.
- Known drift risks such as enum expansion, nullable fields, field removals,
  version fall-forward, auth-policy changes, rate limits, and webhook payload
  versioning.

## Files To Replace After Copying

- `package.json`: package name, version, service SDK dependencies, scripts if a
  provider needs extra deterministic test setup.
- `.env.example`: provider env vars, safe placeholders, required scopes, and OTEL
  examples.
- `README.md`: current connector scope, env setup, runtime wiring, webhook setup,
  testing, and tracing.
- `AGENTS.md`: provider-wide docs, versioning, auth, pagination, webhooks,
  resource map, current implementation, drift risks, code map, and verification.
- `src/schemas.ts`: replace template `Post` and webhook schemas with schemas
  derived from real recorded payloads or fixtures.
- `src/api.ts`: replace JSONPlaceholder client with provider auth, base URL,
  pagination, response decoding, and typed errors.
- `src/streams.ts`: update entity stream names, cursor fields, cutoff logic, and
  page transition behavior.
- `src/connector.ts`: rename service/config types, define entities, wire streams,
  implement provider webhook verification, and route documented event types.
- `src/sandbox.ts`: rename service name, env vars, port, runtime layers, and
  provider-specific telemetry redaction.
- `src/index.ts`: export public namespaces and schemas.
- `test/api.vcr.test.ts`: record/replay real provider API behavior for REST or
  GraphQL connectors.
- `test/webhook.test.ts`: add deterministic webhook fixtures and verification
  tests, including signature failures.
- `test/helpers.ts`: update config layers and test helpers.
- `test/__cassettes__/`: generated by VCR only; never hand-edit.

## Current Patterns To Preserve

- Use `Context.Service` for connector and API client services.
- Prefer `make`, `layer(config)`, and `layerConfig(Config.Wrap<...>)` naming.
- Use `Config`/`ConfigProvider.fromEnv()` for runtime config. Do not read
  `process.env` in connector code.
- Keep one final `Effect.provide(...)` in runnable entrypoints where practical.
- Use `Layer.provide(...)` to satisfy layer dependencies before `Layer.mergeAll`.
- Use `Webhook.route(...)` for webhook routes and schema-validated payloads.
- Signed webhook verification must fail closed if required raw-body or signature
  inputs are missing.
- Use `Ingestion.runConnector(...)` for sandbox/runtime wiring.
- Use `Telemetry.layerOtlpTracing()` for optional trace export in sandboxes.
- Add provider-specific `redactedHeaders` for custom secret headers.
- Keep logs local via the sandbox logger; telemetry export is traces only by
  default.
- Use deterministic VCR replay for REST/GraphQL API tests and deterministic
  fixtures or mock servers for non-HTTP protocols.

## Template Current Implementation

- Package: `@useairfoil/producer-template`.
- API base URL default: `https://jsonplaceholder.typicode.com`.
- Entity: `posts`.
- Webhook path: `/webhooks/template`.
- Pagination style: `_page` and `_limit` query parameters.
- Webhook signature hook is a placeholder that accepts events. Replace it for any
  real provider that signs webhooks.
- Sandbox telemetry uses `Telemetry.layerOtlpTracing()` with Connector Kit default
  sensitive-header redaction.

## Safety Rules

- Do not hand-edit VCR cassettes.
- Do not commit `.env` or provider secrets when adapting the template.
- Do not make schemas broad to pass tests; schema flexibility must be backed by
  provider docs, changelog, cassettes, or fixtures.
- Do not add backward-compatibility shims unless persisted data, shipped behavior,
  external consumers, or explicit requirements need them.
